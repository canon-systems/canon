# Complete Schema Migration

## Migration: 20250102_complete_schema_migration.sql

### Purpose
This is a comprehensive migration that sets up all database objects for the new schema:
- **Functions**: All database functions needed for the new schema
- **RLS Policies**: Row-level security policies for all tables
- **Primary/Foreign Keys**: All constraints ensuring data integrity
- **Indexes**: Performance indexes for all tables
- **Triggers**: Automated triggers for timestamp updates

### What This Migration Does

#### 1. Functions
- **`get_next_document_version(doc_id)`**: Gets the next version number for a document
- **`upsert_repo_file_summary(...)`**: Upserts file summaries (updated to remove submissions reference)
- **`update_updated_at_column()`**: Trigger function to auto-update timestamps

#### 2. Primary Keys & Constraints
- Ensures primary keys exist on all tables
- Creates unique constraints where needed (e.g., `document_files(document_id, file_path)`)

#### 3. Foreign Keys
- `documents.repo_id` → `workspace_repos.id`
- `document_files.document_id` → `documents.id`
- `document_versions.document_id` → `documents.id`
- `repository_setup.repo_id` → `workspace_repos.id`
- `architecture_diagrams.repo_id` → `workspace_repos.id`

#### 4. Indexes
Creates performance indexes on:
- `documents`: repo_id, created_at, updated_at
- `document_files`: document_id, file_path, composite
- `document_versions`: document_id, version_number, created_at
- `repository_setup`: repo_id, setup_status
- `repo_file_summaries`: repo_id, file_path, branch, composite
- `workspace_repos`: workspace_id, repo_url

#### 5. Row-Level Security (RLS)
Enables RLS and creates policies for:
- **documents**: Users can only access documents for repos they own
- **document_files**: Users can only access files for their documents
- **document_versions**: Users can only access versions for their documents
- **workspace_repos**: Users can only access their own repos
- **repository_setup**: Users can only access setup for their repos
- **repo_file_summaries**: Users can read summaries for their repos; service role can manage

#### 6. Triggers
- Auto-updates `updated_at` timestamp on `documents` table when records are updated

#### 7. Data Integrity
- Removes orphaned `document_files` records
- Removes orphaned `document_versions` records
- (Orphaned documents cleanup is commented out for safety)

### How to Run

#### Option 1: Using Supabase Dashboard
1. Go to your Supabase project dashboard
2. Navigate to **SQL Editor**
3. Copy the contents of `20250102_complete_schema_migration.sql`
4. Paste and execute

#### Option 2: Using Supabase CLI
```bash
# If you have Supabase CLI installed
supabase db push
```

#### Option 3: Using psql
```bash
psql -h <your-db-host> -U <your-user> -d <your-database> -f 20250102_complete_schema_migration.sql
```

### Important Notes

1. **Idempotent**: This migration is designed to be safe to run multiple times. It uses `IF NOT EXISTS` and `DROP IF EXISTS` patterns.

2. **RLS Policies**: The migration drops and recreates all RLS policies. If you have custom policies, you may need to add them back after running this migration.

3. **Service Role**: The `upsert_repo_file_summary` function uses `SECURITY DEFINER` to allow service role access. Make sure your service role has proper permissions.

4. **Data Safety**: The migration includes cleanup of orphaned records, but deletion of orphaned documents is commented out. Review and uncomment if needed.

### Verification

After running the migration, you can verify everything is set up correctly by running the verification queries at the bottom of the migration file (they're commented out).

### Dependencies

This migration assumes:
- The new tables (`documents`, `document_files`, `document_versions`, `workspace_repos`, etc.) already exist
- You've migrated your data from the old schema to the new schema
- You have proper authentication set up (auth.uid() function available)

### Troubleshooting

If you encounter errors:

1. **"relation does not exist"**: Make sure all tables exist before running this migration
2. **"permission denied"**: Make sure you're running as a user with sufficient privileges
3. **"constraint already exists"**: This is normal if you've run parts of the migration before. The migration handles this gracefully.
